  <name>Task 1: Create pipeline configuration and PipelineModeResolver</name>
  <files>
    blackbox/backend/blackbox_service/src/main/java/com/cgi/blackbox/service/implementation/PipelineConfig.java,

    blackbox/.env

  <action>
    1. Create PipelineConfig.java — a @ConfigurationProperties class bound to prefix "pipeline":
       - `mode` (String, default "auto") — values: auto, transcription_only, transcription_diarization, full
       - `healthCheck.diarizationUrl` (String, default "http://localhost:8082/health")
       - `healthCheck.askChatUrl` (String, default "http://localhost:8083/health")
       - `healthCheck.timeoutMs` (int, default 3000)

    2. Create PipelineModeResolver.java — a @Component that resolves the actual pipeline mode:
       - Constructor-inject PipelineConfig and a RestTemplate (create a @Bean for RestTemplate in a config class or use RestTemplateBuilder — keep it simple, a new RestTemplate() in the constructor is fine for this use case)
       - `resolve()` method:
         - If configuredMode is NOT "auto": call `verifyAndWarn(configuredMode)` which checks that required services are healthy. If not, log a WARN with the specific services that are down and downgrade to the best available mode. Return the resolved mode.
         - If configuredMode IS "auto": check diarization health, check chat health. If both up → "full". If only diarization → "transcription_diarization". Otherwise → "transcription_only".
       - `checkHealth(String url)` private method: GET request with configured timeout. Return true if 2xx, false on any exception. Log at DEBUG level.
       - Add a @PostConstruct method that:
         - Detects system RAM using com.sun.management.OperatingSystemMXBean.getTotalMemorySize()
         - Logs RAM-based mode recommendation at INFO level (< 8GB: transcription_only recommended, < 16GB: transcription_diarization recommended, >= 16GB: full supported)
         - Logs the configured pipeline mode
         - Does NOT resolve mode at startup (health checks happen at trigger time per RESEARCH.md pitfall #2)

    3. Add to application.yml under existing config:
       ```yaml
       pipeline:
         mode: ${PIPELINE_MODE:auto}
         health-check:
           diarization-url: ${BACKEND_DIARIZATION_BASE_URL:http://localhost:8082}/health
           ask-chat-url: ${BACKEND_ASK_CHAT_BASE_URL:http://localhost:8083}/health
           timeout-ms: 3000
       ```

    4. Add to blackbox/.env:
       ```
       PIPELINE_MODE=auto
       BACKEND_DIARIZATION_BASE_URL=http://localhost:8082
       ```

    AVOID: Don't use Spring Boot Actuator for the backend's own health — it's not needed here. We're just calling the Python services' /health endpoints. Don't check health at startup for mode resolution (per RESEARCH.md pitfall #2 — diarization service loads ML models slowly). The @PostConstruct only logs RAM recommendation and configured mode.

    NOTE: Ensure @ConfigurationProperties is enabled. Check if @EnableConfigurationProperties is needed or if component scanning picks it up. Using @ConfigurationProperties(prefix = "pipeline") with @Component is the simplest approach.
  </action>
