  <files>

  <action>
    Create PipelineOrchestrationService that replaces the Conductor workflow with direct orchestration:

    1. Read the existing workflow JSON (builtin-transcription-workflow.json) to understand the exact flow:
       - transcribe (call diarization service REST endpoint)
       - save_transcription (call existing persistence logic)
       - SWITCH on pipeline_mode:
         - "full": stringify_transcription → ask_chat(determine_speakers) → save_determined_speakers → combine_transcription_with_round_definition → ask_chat(determine_rounds) → save_determined_rounds → execute_prompts
         - "transcription_diarization": stringify_transcription only
         - "transcription_only": same as transcription_diarization

    2. Implement as a Spring @Service with an async method (@Async or CompletableFuture) that:
       - Takes workspace_id, file_urls, pipeline_mode, language, round_definition as parameters
       - Calls diarization service HTTP endpoint for transcription (RestTemplate/WebClient)
       - Calls persistence layer to save transcription results
       - Branches by pipeline_mode (simple if/else, NOT Conductor SWITCH)
       - For "full" mode: calls ask-chat-service HTTP endpoint for LLM tasks, saves results
       - Broadcasts progress via SSE (reuse existing SSEHandler pattern)

    3. Read each Java task worker (SaveTranscriptionTask, SaveDeterminedSpeakersTask, SaveDeterminedRoundsTask, ExecutePromptsTask) and inline their logic into the orchestration service. These tasks are simple — they extract input params, call a service method, and return output. The orchestration service should call the same service methods directly.

    4. Read the two INLINE JavaScript tasks in the workflow JSON. These do data transformation (stringify transcription, combine with round definition). Reimplement as Java methods in the orchestration service.

    5. Update AnalysisServiceImpl.runAnalysisWorkflow() to call PipelineOrchestrationService instead of ConductorWorkflowClient.triggerWorkflow().

    **Important:** Keep the async fire-and-forget pattern — the REST endpoint returns immediately, orchestration runs in background, SSE broadcasts results. Use @Async with a configured TaskExecutor, NOT blocking the request thread.

    **Error handling:** Wrap each step in try/catch. On failure, broadcast error via SSE (same as current behavior when Conductor tasks fail). Include retry logic for HTTP calls to Python services (3 retries with 2s backoff) — Conductor provided this transparently.

    **Avoid:** Don't introduce a new workflow engine or state machine library. The logic is simple enough for direct Java code. Don't change the REST API contract — frontend should see no difference.
  </action>
  <verify>
    `cd blackbox && mvn clean install -DskipTests` compiles without errors.
    Grep for ConductorWorkflowClient usage — should only remain in import/interface (not called from AnalysisServiceImpl).
  </verify>

    All 4 Java task worker logic inlined into orchestration flow.
