  <action>
    1. Read ALL data object files thoroughly first. Understand the nesting structure:
       - WorkspaceDO: contains TranscriptionDO (embedded), List<FragmentDO>, List<DeterminedRoundDO> (each with List<PromptResponseDO>), List<DeterminedSpeakerDO>
       - WargameSetupDO: contains List<PromptDO>

    2. Strategy for nested documents: Use JPA @ElementCollection and @Embeddable for nested objects. H2 supports this well. The nested objects (FragmentDO, DeterminedRoundDO, etc.) become @Embeddable classes stored in separate collection tables with foreign keys. This preserves the object structure without needing to flatten.

       Alternative if @ElementCollection gets too complex: Use @Lob with JSON serialization (Jackson) for deeply nested fields. H2 supports CLOB columns that can store JSON strings. Read/write via @Convert(converter = JsonConverter.class).

    3. Update pom.xml:
       - Remove: spring-boot-starter-data-mongodb, testcontainers mongodb
       - Add: spring-boot-starter-data-jpa, com.h2database:h2 (runtime scope)

    4. Replace MongoConfiguration.java with JpaConfiguration.java:
       - @Configuration @EnableJpaRepositories
       - H2 datasource configuration (file-based: jdbc:h2:file:./data/blackbox-db)
       - Use file-based H2 (not in-memory) so data persists across restarts

    5. Update repository interfaces:
       - IBaseRepository: extend JpaRepository<O, String> instead of MongoRepository
       - Remove the @Query with MongoDB JSON syntax, replace with @Query JPQL or Spring Data method name query
       - WorkspaceRepository and WargameSetupRepository: change extends to IBaseRepository (should work if IBaseRepository changes)

    6. Update data objects:
       - AbstractDO: Add @MappedSuperclass, @Id with @GeneratedValue(strategy = GenerationType.UUID) or keep String ID
       - WorkspaceDO: Add @Entity, @Table. Convert nested objects to @Embeddable/@ElementCollection
       - WargameSetupDO: Add @Entity, @Table. Convert PromptDO list to @ElementCollection

    7. Read AbstractGenericAccess.java — it uses MongoOperations for custom queries. Replace with EntityManager or JPA Criteria API. Keep the same method signatures.

    8. Update application.yml:
       - Remove MongoDB connection config
       - Add H2 datasource config:
         spring.datasource.url: jdbc:h2:file:./data/blackbox-db
         spring.datasource.driver-class-name: org.h2.Driver
         spring.jpa.hibernate.ddl-auto: update
         spring.jpa.database-platform: org.hibernate.dialect.H2Dialect

    **Important:** The package path currently contains "mongodb" (com.cgi.blackbox.persistance.mongodb). Either rename the packages (cleaner but more churn) or leave them (less churn, misleading name). Prefer renaming since this is a major migration and the old name would be confusing. Rename "mongodb" to "jpa" or "persistence" in the package structure.

    **Avoid:** Don't try to replicate MongoDB's flexible schema. Use proper JPA entity modeling. Don't use in-memory H2 — use file-based for data persistence.
  </action>

  </verify>

  <name>Task 2: Replace GridFS with filesystem storage</name>
  <files>
    blackbox/backend/blackbox_persistance/src/main/java/com/cgi/blackbox/persistance/mongodb/implementation/WorkspaceMediaAccess.java,
    blackbox/backend/blackbox_persistance/src/main/java/com/cgi/blackbox/persistance/mongodb/implementation/WargameSetupRulebookAccess.java,
    blackbox/backend/blackbox_application/src/main/resources/application.yml
  </files>
  <action>
    1. Read WorkspaceMediaAccess.java thoroughly. It uses GridFsOperations for:
       - listAllMediaFiles(workspaceId) — list files by metadata
       - saveMediaFile(workspaceId, media, inputData) — store with metadata
       - peekMediaFile(workspaceId, mediaId) — get metadata
       - openMediaFile(workspaceId, mediaId, outputStream) — stream content
       - deleteMediaFile(workspaceId, mediaId) — delete

    2. Read WargameSetupRulebookAccess.java. Similar pattern for rulebook files.

    3. Create a FileStorageService that stores files on the local filesystem:
       - Base directory: configurable via application.yml property (e.g., `app.storage.base-path: ./data/files`)
       - Directory structure: `{base}/media/{workspaceId}/{mediaId}/{filename}` and `{base}/rulebooks/{wargameSetupId}/{filename}`
       - Metadata: Store as JSON sidecar file next to each stored file (e.g., `{filename}.meta.json`)
       - Use java.nio.file.Files for all operations (portable, handles paths correctly)

    4. Rewrite WorkspaceMediaAccess to use FileStorageService instead of GridFsOperations:
       - listAllMediaFiles: list files in workspace directory
       - saveMediaFile: write to filesystem + metadata JSON
       - peekMediaFile: read metadata JSON
       - openMediaFile: stream from filesystem (Files.copy to OutputStream)
       - deleteMediaFile: delete file + metadata

    5. Rewrite WargameSetupRulebookAccess similarly.

    6. Generate unique IDs for new files using UUID.randomUUID().toString() (replacing MongoDB ObjectId).

    7. Add storage configuration to application.yml:
       ```
       app:
         storage:
           base-path: ./data/files
       ```

    **Important:** Create the data/files directory on application startup if it doesn't exist (use @PostConstruct or ApplicationRunner).

    **Avoid:** Don't use database BLOBs — filesystem is simpler and more portable for large audio files. Don't change the service interface contracts (the layer above shouldn't know storage changed).
  </action>
