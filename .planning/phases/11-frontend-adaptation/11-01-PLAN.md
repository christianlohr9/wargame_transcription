  <name>Task 1: Create PipelineStatusAPI REST endpoint</name>
  <files>
    blackbox/backend/blackbox_application/src/main/java/com/cgi/blackbox/rest/PipelineStatusAPI.java,
    blackbox/backend/blackbox_application/src/main/java/com/cgi/blackbox/rest/dtos/PipelineStatusDto.java

  <action>
Create a new REST controller `PipelineStatusAPI` with a single GET `/pipeline/status` endpoint that returns the current pipeline status.

Create `PipelineStatusDto` with fields:
- `String resolvedMode` — result of PipelineModeResolver.resolve()
- `String configuredMode` — raw value from PipelineConfig.getMode()
- `Map<String, Boolean> services` — map with keys "diarization" and "askChat", values from health checks

Inject `PipelineModeResolver` and `PipelineConfig`. For service health, reuse the health check logic already in PipelineModeResolver — extract the checkHealth method to be package-accessible (not private) so the controller can call it, OR add a method to PipelineModeResolver that returns the service status map.

Prefer adding a `getServiceHealth()` method to PipelineModeResolver that returns `Map<String, Boolean>` with "diarization" and "askChat" keys — this keeps the health check logic centralized. The resolve() method already calls checkHealth internally, so this is just exposing it cleanly.

Do NOT cache the health check results — each call to /pipeline/status should check live health. The health checks have a 3-second timeout configured in PipelineConfig, which is acceptable for an on-demand status check.

Return the DTO directly from the GET endpoint (Spring Boot auto-serializes to JSON).
  </action>
  <verify>
The project should compile. Verify the endpoint structure is correct by reviewing the file. The endpoint will return JSON like:
```json
{
  "resolvedMode": "transcription_only",
  "configuredMode": "auto",
  "services": {
    "diarization": false,
    "askChat": false
  }
}
```
  </verify>
  <done>
GET /pipeline/status endpoint exists, returns resolved mode, configured mode, and live service health status as JSON.
  </done>

<task type="auto">
  <name>Task 2: Add service status indicator to AppBar</name>
  <files>
    blackbox/frontend/src/components/AppBar.vue,
    blackbox/frontend/src/components/models.ts
  </files>
  <action>
Add a `PipelineStatus` interface to models.ts:
```typescript
export interface PipelineStatus {
  resolvedMode: string
  configuredMode: string
  services: Record<string, boolean>
}
```

In AppBar.vue, add a subtle status indicator to the right side of the toolbar (before the settings button):

1. On mount, fetch `GET /pipeline/status` via the existing `api` axios instance. Store in a `pipelineStatus` ref.

2. Display a compact status area with small colored dots:
   - Green dot + "Diarization" if services.diarization is true
   - Green dot + "Chat" if services.askChat is true
   - Grey dot + strikethrough text for services that are down
   - Show the resolved mode as a small label (e.g., "Mode: full" or "Mode: transcription only")

3. Use Quasar's `q-chip` components for a clean, subtle look — small size, outline style. Group them in a row with `q-space` before them to push right.

4. Format mode names for display: replace underscores with spaces, title case (e.g., "transcription_only" → "Transcription Only").

5. Add a refresh capability — a small `q-btn` with a refresh icon that re-fetches the status. This lets the user check again after starting a service.

6. Handle the fetch error gracefully — if the backend is unreachable, show a single grey chip "Backend unavailable". Do not show error dialogs.

Keep the indicator compact — it should not dominate the toolbar. The existing toolbar has: logo, title, workspace selector, add/delete buttons, settings button. Place the status chips between the workspace controls and the settings button, separated by a `q-space`.
  </action>
  <verify>Run `cd blackbox/frontend && npx vue-tsc --noEmit` to check for TypeScript errors. If vue-tsc is not available, check that `npm run lint` passes.</verify>
  <done>
AppBar shows colored service status chips indicating which services are available, the resolved pipeline mode, and a refresh button. Backend unavailable state handled gracefully.
  </done>
</task>
